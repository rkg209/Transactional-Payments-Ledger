# ADR 0002 — jOOQ code-generation mechanism

Date: 2026-07-15
Status: accepted
Deciders: Project owner
Relates to: SPEC 0001, CON-5 (all SQL through jOOQ)

## Context

CLAUDE.md and SPEC 0001 both require type-safe jOOQ repositories over *generated* code
(`org.ledger.db.generated`), not the plain-SQL `DSL.field(...)`/`DSL.table(...)` API the SPEC 0000
walking skeleton used as a bootstrap shortcut. Generating that code requires introspecting a real
Postgres catalog that already has the schema applied — the generator needs a live database to point
at, not just the migration files.

Several mechanisms exist to get jOOQ a database to introspect at build time, and they differ in how
faithfully they reproduce the real schema (triggers, CHECK constraints with regexes, partial
indexes) and in what they add to the build's dependency surface.

## Options considered

### Option A — jOOQ's own `jooq-meta-extensions` XML/DDL parser (no live DB)
- **Pros:** No Docker dependency for codegen; fast; works offline.
- **Cons:** Re-implements a subset of Postgres DDL parsing — it does not run `plpgsql`, so the
  `ledger_entries_immutable_tg` trigger and its function are invisible to it (harmless for codegen,
  since triggers don't produce generated types, but it signals the tool isn't looking at the real
  engine). More importantly, some CHECK constraint syntax (regex `~` operators, `now()` defaults) has
  historically tripped up the DDL parser's grammar, which is a maintenance tax with no offsetting
  benefit here, since Docker is already a hard build dependency for the Testcontainers integration
  tests.

### Option B — a manually-managed long-lived dev Postgres + `jooq-codegen-maven` pointed at it
- **Pros:** Fast incremental runs (no container startup per build); conventional Maven jOOQ setup.
- **Cons:** Requires a developer (or CI) to remember to keep a database in the right migrated state
  before every codegen run — an out-of-band, easy-to-forget precondition. Codegen output would
  silently drift from the migrations whenever someone edits a migration and forgets to re-migrate the
  dev DB. That is exactly the class of "it worked on my machine" bug this project is trying to
  eliminate by construction.

### Option C — `testcontainers-jooq-codegen-maven-plugin`, bound to `generate-sources` (chosen)
- **Pros:** Starts a real `postgres:16-alpine` container, runs the actual Flyway migrations against
  it, introspects the live catalog, generates code, tears the container down — every codegen run is
  from a known-clean, fully-migrated schema, so generated code can never drift from the migrations
  that produced it. Reuses the same Postgres image and Testcontainers machinery the integration tests
  already depend on, so it adds no new class of infrastructure dependency. Fidelity is total: regex
  CHECKs, `plpgsql` triggers, partial indexes are all exactly what production will see, because it's
  the same engine.
- **Cons:** Every clean build now needs Docker, even for a developer who only wants to compile
  (`generate-sources` runs before `compile`). Adds meaningfully to `mvn clean install` wall-clock time
  (container startup on every fresh build). Couples the *build* to Docker availability, not just the
  test suite, which is a strictly larger blast radius if Docker is ever unavailable in some future
  build environment.

## Decision

**Option C.** Use `testcontainers-jooq-codegen-maven-plugin` bound to the `generate-sources` phase:
`database.type=POSTGRES`, `containerImage=${postgres.image}` (the same `postgres:16-alpine` pin used
everywhere else), `flyway.locations=filesystem:src/main/resources/db/migration`,
`inputSchema=public`, `excludes=flyway_schema_history`, output package `org.ledger.db.generated` into
`target/generated-sources/jooq`. Generated code is **never committed** — `target/` is regenerated by
every build, exactly like `WalkingSkeletonIT`'s Testcontainers use already assumes Docker is present.

Docker is already an unconditional hard dependency for this project (the concurrency and crash
harness — SPEC 0007 — cannot exist without it), so Option C adds no new category of risk; it only
moves an existing dependency earlier in the build lifecycle.

### The Docker-API-pin propagation problem

`pom.xml` already pins `-Dapi.version=1.44` via `systemPropertyVariables` on surefire and failsafe,
because the local Docker Engine (API 1.55) rejects the docker-java client's default handshake
(API 1.32) with an HTTP 400 that looks like "no Docker found" (see `progress_report.md` [002]).
Codegen runs Testcontainers in the **Maven JVM itself** at `generate-sources` — not inside a
surefire/failsafe fork — so `systemPropertyVariables` on those plugins never reaches it. Without a
fix, codegen fails with the same misleading "Docker not found" symptom the earlier fix already
diagnosed once.

The fix is `.mvn/jvm.config` containing `-Dapi.version=1.44`: Maven reads this file and applies the
system property to its own JVM before any plugin — including the codegen plugin — runs, so every
Testcontainers consumer in the build (Maven-JVM codegen, surefire forks, failsafe forks) negotiates
the same pinned API version from one place.

## Consequences

**Positive:**
- Generated code can never drift from the migrations — it is regenerated from the real, migrated
  schema on every build, by construction, not by developer discipline.
- No new infrastructure dependency class: Docker was already required.
- One `.mvn/jvm.config` fixes API-pin propagation for every current and future Testcontainers
  consumer in the build, not just the two plugins it was originally added for.

**Negative / accepted costs:**
- Every clean build requires Docker, including a bare `mvn compile`. There is no "just compile, no
  Docker" escape hatch in this design.
- Container startup adds real wall-clock time to `generate-sources` on every clean build (mitigated
  only by Maven's incremental build skipping recompilation when generated sources are unchanged
  within a single `target/`, which does not help a truly clean tree).
- Generated sources are gitignored and must be regenerated after `git clone` before the IDE can
  resolve `org.ledger.db.generated` types — a first-run surprise if undocumented (documented here and
  in `progress_report.md`).

**What would change our mind:**
- If Docker ever becomes unavailable in a build environment this project must support (e.g. some
  future air-gapped CI), Option A becomes necessary despite its fidelity gap, and would need to be
  paired with a scheduled job that diffs its output against a real-Postgres introspection to catch
  drift.

## Addendum — plugin classpath conflicts (discovered during implementation)

`testcontainers-jooq-codegen-maven-plugin:0.0.4`'s own POM pins old transitive versions (jOOQ
3.18.3, Testcontainers 1.19.1, Flyway 9.16.3, Jackson 2.10.3) that do not match the versions this
project's `pom.xml` pins for its runtime dependencies (Flyway 10.10.0 via Spring Boot 3.3.5's BOM,
Jackson 2.15.2). The plugin resolves its own execution classpath with a self-first classloading
strategy, so declaring our newer `flyway-core`/`flyway-database-postgresql` directly in the
plugin's `<dependencies>` block was not enough on its own — the plugin's transitively-pulled
`flyway-mysql:9.16.3` (compiled against Flyway 9.x's API, unused since `database.type=POSTGRES`)
and `jackson-annotations:2.10.3` remained on the classpath and were binary-incompatible with
Flyway 10.x / Jackson 2.15.x, producing `NoClassDefFoundError`s
(`FlywayTeamsUpgradeRequiredException`, then `JsonKey`) that looked like missing dependencies
rather than version conflicts.

Fix: explicitly pin `flyway-mysql` and `jackson-annotations` to the same versions as the rest of
the toolchain (`${flyway.version}`, `2.15.2`) directly in the plugin's `<dependencies>` block, even
though `flyway-mysql` is never exercised. This is a maintenance tax: every time `flyway.version` or
Spring Boot's Jackson version moves, these two pins need to move with them, or the same class of
failure resurfaces silently. Documented here so the next version bump doesn't have to
re-diagnose it from scratch.
